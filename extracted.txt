CHAPTER 1
INTRODUCTION
1.1 Project Description
Modern electronic commerce applications display massive catalogues of products to online shoppers. Consumers browse these options to discover items matching their personal tastes and current requirements. Personalised recommendation services serve as the primary mechanism to filter these diverse options.
Standard recommending frameworks fail when users browse across distinct mobile and desktop devices. This digital fragmentation creates separate profiles for the same physical customer in the database. Such disconnection prevents the delivery of consistent recommendations across the consumer purchase journey.
Furthermore, traditional services continue showing ads for items that customers have already purchased. This occurs because the underlying software lacks awareness of the product usage life cycle. Suggesting identical items immediately after purchase causes consumer irritation and recommendation fatigue.
The Contextual Product Recommender Platform represents a decoupled microservices architecture designed for e-commerce. This platform resolves identity fragmentation by merging multiple device sessions into a single profile. It implements temporal suppression rules to hide recently bought items for realistic periods.
The platform uses a hybrid machine learning engine to suggest highly relevant items. It integrates content similarity, collaborative filtering and semantic embeddings into one score. All platform services run inside containers to ensure easy deployment in cloud environments.

2.2 Gaps Identified
The study of existing academic literature revealed four primary limitations
Existing recommendation architectures do not resolve multiple device logins into a unified user profile [9]
Current pipelines rely on batch computations which cannot adapt immediately to live clickstreams [17]
Recommending models lack feedback-aware learning which continues displaying recently purchased consumable categories [13]
No public platform integrates streaming ingestion with five-layer database stores and automated suppression
2.3 Proposed System
The proposed platform provides an end-to-end personalization framework that resolves the gaps. It processes 
live clickstreams using an asynchronous queue to avoid blocking storefront requests 
identity mapping table to resolve diverse session logs to one unique core customer record 
product lifetime database to automatically suppress purchased categories based on average usage timelines 
hybrid machine learning engine to produce ranked recommendations using content, collaborative and semantic signals
Problem Statement
To develop a platform that unifies user identities to build dynamic user profiles, generates personalized recommendations and continuously updates the profiles based on real-time user feedback to refine future recommendations.
Objectives
To unify user interaction data from digital platforms into a single, centralized user profile by resolving fragmented identities across devices and sessions.
To develop dynamic user profiles by analyzing contextual signals and evolving interest patterns, enabling the platform to learn user habits and predict future interests.
To implement a personalised recommendation service using content-based filtering that adapts in real time based on both positive and negative user feedback.
The engine exposes FastAPI endpoints for ingestion and recommendation retrieval. It coordinates streaming tasks via an Apache Kafka pipeline running in the background. An administration dashboard provides real-time monitoring of active suppressions and database health.
2.4 Tools and Technologies Used
The platform relies on several open-source technologies to achieve low latency. FastAPI serves as the API gateway because it provides fast asynchronous request handling. Apache Kafka decouples the incoming web events from the downstream database write queries. MySQL manages the relational schemas required for cross-device identity mapping and profiling. The machine learning models are built using scikit-learn, rank-bm25 and sentence-transformers libraries [19].

CHAPTER 3
SOFTWARE REQUIREMENT SPECIFICATION
3.1 Introduction
This chapter details the requirements for the Contextual Product Recommender Platform. These specifications define how the platform ingests events, resolves identities and serving recommendations. They establish the boundaries for performance, security, database structure and integration methods.
3.2 General Description
The platform operates as a standalone service accessible via REST API endpoints. E-commerce websites capture clickstreams and forward them to this API without blocking browsers. The database stores profile records across five functional layers to support complex queries. The design focuses on decoupling ingestion pipelines from recommendation delivery pipelines.
3.3 Functional Requirements
FR-1: Asynchronous Event Ingestion:The platform shall ingest clickstream events asynchronously using a Kafka queue to handle high traffic without blocking the main application.
FR-2: Unified Identity Resolution: The identity module shall map all incoming user session identifiers to a single core_id UUID for consistent, persistent tracking.
FR-3: Multi-Layer Customer Profiling: The database shall maintain a five-layer customer profile to store attributes, preferences and historical interactions for real-time personalisation.
FR-4: Affinity Score Calculation: The engine shall calculate category affinity scores by assigning varying mathematical weights to specific user interactions.
FR-5: Purchase Lifecycle Suppression: The suppression module shall exclude recently purchased categories based on lifecycle durations to prevent recommending items the user just bought.
FR-6: Hybrid Recommendation Engine: The engine shall merge BM25, ALS and deep learning embedding signals to accurately rank and suggest highly relevant products.
FR-7: Cold-Start User Handling: The service shall provide category-based recommendations using broad popularity signals for new users lacking historical data.
FR-8: Administrative Profiling Dashboard: An interface shall display live profiling metrics and active queues for administrators to monitor system health and real-time activity.
FR-9: Evaluation Metrics API: The API shall expose endpoints for standard retrieval metrics to continuously monitor model accuracy.
FR-10: Stale Interest Decay: A daily database job shall apply exponential decay to stale interest scores, ensuring recommendations heavily prioritize recent preferences.
3.4 External Interface Requirements
The platform communicates with several external services and database systems. It exposes FastAPI HTTP endpoints to accept JSON event payloads from the storefront. It relies on the Apache Kafka driver to publish messages to the streaming queue. The database connections are managed via secure MySQL connectors using pool objects. Optionally, the notification service connects to the SendGrid Web API to send emails.
3.5 Non-Functional Requirements
NFR-1: Real-Time Latency: The system shall return personalised product recommendations within a maximum latency of 200 milliseconds per request.
NFR-2: System Scalability: The platform shall scale horizontally to support high-throughput traffic spikes, facilitated by Kafka and independent microservices.
NFR-3: Data Security and Privacy: All user profile data must be encrypted at rest and in transit and personally identifiable information (PII) must be anonymised.
NFR-4: High Availability: Core services and databases shall maintain 99.9% uptime by deploying across multiple availability zones with automated failover.
NFR-5: Maintainability and Observability: The system must implement comprehensive logging and expose metrics to a central monitoring system for rapid troubleshooting.
3.6 Design Constraints
The platform relies on pre-computed similarity matrices which limits dynamic catalog updates. Adding new products requires executing offline retraining scripts to update similarities. The database architecture requires MySQL transactions which limits horizontal scaling of writes. All Python services must target version 3.10 or higher due to library dependencies.
3.7  Other Requirements
A database configuration file must specify the credentials and connection limits. Environmental variables must define the host addresses of the database and Kafka. The model files must be present in the designated folder before starting services.

4.1 System Architectural Design
The High-Level Platform Deployment Architecture illustrates a decoupled, modular system structured into distinct layers. This separation ensures that high-volume data ingestion tasks do not degrade the read performance of recommendation queries.
Client & API Gateway Layers: Manage user interactions, request routing and secure access to the platform's endpoints.
Application Services Layer: Strictly separates the write-heavy Ingestion Service from the read-heavy Recommendation Engine.
Message Broker Layer: Asynchronously queues incoming behavioral events, preventing database bottlenecks during sudden traffic spikes.
Data Persistence Layer: Utilizes a tailored storage strategy, combining a relational database for dynamic user profiles, an in-memory cache for rapid recommendation delivery and data lakes for storing historical event logs used in model training.

CHAPTER 5
IMPLEMENTATION
5.1 Code Snippets
The following pseudocode defines the algorithms used in the platform execution:
Algorithm 1: Cross-Device Identity Resolution Module
This module solves the issue of fragmented user identities across different devices. It takes an incoming identifier and checks if it exists in the database. If found, it links the current session to the master core_id. If not, it generates a brand-new unique user profile and securely maps the new device identifier to it, ensuring a continuous, unified shopping experience.
Begin
procedure RESOLVE_IDENTITY(identifier_type, identifier_value)
    query = "SELECT core_id FROM identities WHERE type = ? AND value = ?"
    record = db.execute(query, identifier_type, identifier_value)
    if record exists then
        return record.core_id
    else
        new_uuid = generate_uuid_v4()
        db.execute("INSERT INTO users (core_id) VALUES (?)", new_uuid)
        db.execute("INSERT INTO identities (core_id, type, value) VALUES (?, ?, ?)", 
                   new_uuid, identifier_type, identifier_value)
        return new_uuid
    end if
end procedure
End
Algorithm 2: Profile Update and suppression Module
This algorithm dynamically updates a user's interest profile based on their real-time actions. It calculates a score using predefined weights. Most importantly, if the interaction is a "purchase", it queries the expected usage lifetime of that product and sets a suppress_until date. This completely prevents the system from annoying the user by recommending an identical consumable item until they are likely to need a refill.
Begin
procedure UPDATE_PROFILE(core_id, category, brand, event_type)
    weight = EVENT_WEIGHTS[event_type]
    profile = db.query("SELECT * FROM interest_profiles WHERE core_id = ? AND category = ?", 
                       core_id, category)
    if profile exists then
        new_score = max(0, profile.score + weight)
        db.execute("UPDATE interest_profiles SET score = ? WHERE id = ?", new_score, profile.id)
    else
        db.execute("INSERT INTO interest_profiles (core_id, category, brand, score) VALUES (?, ?, ?, ?)",
                   core_id, category, brand, max(0, weight))
    end if
    if event_type == 'purchase' then
        duration = db.query("SELECT lifetime FROM product_lifetime WHERE category = ?", category)
        db.execute("UPDATE interest_profiles SET suppress_until = NOW() + INTERVAL ? DAY ", duration)
    end if
end procedure
End
Algorithm 3: Asynchronous Clickstream Event Ingestion
This module acts as the platform's high-speed entry point. When a shopper interacts with the web application, this algorithm intercepts the clickstream data, validates it and rapidly formats it into a standard JSON message. Instead of executing slow database writes, it pushes the message to an asynchronous Apache Kafka queue and immediately returns a success signal to the user's browser. This decoupled approach is what guarantees the platform's low network latency.
Begin
procedure INGEST_EVENT(api_request_payload)
    // Validate incoming payload schema
    if not VALIDATE_SCHEMA(api_request_payload) then
        return HTTP_400_BAD_REQUEST
    end if
    // Construct event record for messaging queue
    kafka_message = {
        "event_id": generate_uuid_v4(),
        "timestamp": current_timestamp(),
        "identifier_type": api_request_payload.id_type,
        "identifier_value": api_request_payload.id_value,
        "event_action": api_request_payload.action,
        "product_id": api_request_payload.product_id,
        "category": api_request_payload.category,
        "brand": api_request_payload.brand
    }
    // Serialize dictionary to JSON byte string
    byte_payload = json_encode(kafka_message)
    // Asynchronously push to Kafka Topic partition using user identifier as hash key
    delivery_status = KAFKA_PRODUCER.send(
        topic="clickstream_events", 
        key=api_request_payload.id_value, 
        value=byte_payload
    )
    if delivery_status.is_successful() then
        return HTTP_202_ACCEPTED
    else
        return HTTP_500_INTERNAL_SERVER_ERROR
    end if
end procedure
End
Algorithm 4: Background Event Consumer Daemon
This algorithm drives the continuous background worker (daemon) that processes all the queued clickstream events. It continuously polls the Kafka topic for new data. For every event, it sequentially triggers the Identity Resolution logic and the Profile Update logic in the background. It employs a robust error-handling block to ensure that if a database connection momentarily drops, the system refuses to commit the queue offset, preventing any valuable user interaction data from being lost.
Begin
procedure CONSUME_EVENTS()
    consumer = KAFKA_CONSUMER.subscribe(topic="clickstream_events")
        while True do
        // Fetch a batch of messages from the topic
        batch_records = consumer.poll(timeout_ms=100)
        for each record in batch_records do
            event = json_decode(record.value)
            try
                // Step 1: Resolve cross-device identity
                core_id = RESOLVE_IDENTITY(event.identifier_type, event.identifier_value)
                // Step 2: Update dynamic interest profile and temporal suppression
                if event.event_action in ['click', 'add_to_cart', 'purchase'] then
                    UPDATE_PROFILE(core_id, event.category, event.brand, event.event_action)
                end if
                // Step 3: Acknowledge successful processing to Kafka broker
          consumer.commit_offset(record.partition, record.offset)
            catch DatabaseConnectionError as e
                log_error("Failed to process event: ", e)
                // Do not commit offset, allowing message to be retried
            end try
        end for
    end while
end procedure
End

Start the database and message broker: Run 'docker-compose up -d' in the terminal.

Train the recommendation model: Run 'python ml/train_hybrid.py' to generate matrices.
Launch the streaming listener: Execute 'python kafka/consumer.py' to process logs.
Run the web API endpoint: Execute 'uvicorn api.app:app --port 5000' to serve recommendations.
Monitor the activity stats: Open 'dashboard/index.html' inside a web browser.

CHAPTER 7
CONCLUSION
The Contextual Product Recommender Platform successfully integrates streaming data ingestion, identity resolution, comprehensive database profiling and hybrid machine learning into a single, unified platform. It effectively resolves the complex cross-device identity mapping challenge by accurately linking deterministic and probabilistic session log determinants. To maintain user engagement, the platform solves the recommendation fatigue problem by utilizing automated category suppression rules that diversify the products shown. Furthermore, its modular, containerized deployment ensures highly efficient data processing and consistently low response latency.
This implementation demonstrates that leveraging open-source tools is a viable, scalable strategy for supporting good personalization services. The platform's hybrid model achieves a high level of precision, which translates directly into meaningful, useful recommendations for customers. Finally, by architecturally decoupling the heavy background ingestion tasks from real-time recommendation lookups, the platform ensures robust, stable performance under varying load
